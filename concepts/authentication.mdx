---
title: "Authentication"
description: "How Ubik Backend authenticates clients using JWT tokens issued by the user management service and validated at the gateway."
---

Ubik uses stateless JWT (JSON Web Token) authentication. The user management service issues signed tokens at login; every subsequent request carries that token in the `Authorization` header. The API gateway validates the token before forwarding traffic to any downstream microservice.

## Auth flow

<Steps>
  <Step title="Login">
    Send a `POST /api/auth/login` request with your credentials.

    ```bash
    curl -X POST https://your-gateway/api/auth/login \
      -H "Content-Type: application/json" \
      -d '{"username": "alice", "password": "s3cr3t"}'
    ```

    A successful response returns a signed JWT:

    ```json
    {
      "token": "eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJhbGljZSIsInJvbGUiOjkxODI3MzY0NTAxOTI4MzcsInVzZXJJZCI6NDIsImlhdCI6MTcwMDAwMDAwMCwiZXhwIjoxNzAwMDg2NDAwfQ.abc123"
    }
    ```
  </Step>
  <Step title="Attach the token">
    Include the token in every subsequent request using the `Bearer` scheme:

    ```bash
    curl https://your-gateway/api/reservations \
      -H "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9..."
    ```
  </Step>
  <Step title="Gateway validates the token">
    `JwtAuthenticationFilter` intercepts every non-public request. It extracts claims from the token, rejects expired or malformed tokens with `401 Unauthorized`, and — on success — mutates the forwarded request by injecting three internal headers:

    | Header | Value |
    |---|---|
    | `X-User-Username` | JWT `sub` claim |
    | `X-User-Role` | JWT `role` claim |
    | `X-User-Id` | JWT `userId` claim |
  </Step>
  <Step title="Downstream services receive enriched headers">
    Each microservice reads `X-User-Username`, `X-User-Role`, and `X-User-Id` directly from the incoming request. They never touch the original JWT.
  </Step>
</Steps>

## Token structure

Tokens are HMAC-SHA256 signed using the shared `JWT_SECRET`. The payload contains:

| Claim | Type | Description |
|---|---|---|
| `sub` | string | Username (e.g., `alice`) |
| `role` | long | Numeric role ID (see [Authorization](/concepts/authorization)) |
| `iat` | long | Issued-at timestamp (Unix seconds) |
| `exp` | long | Expiration timestamp (Unix seconds) |

<Note>
  The `userId` claim is injected by the gateway after it resolves the user record. It is present in the downstream headers but is not part of the raw JWT payload generated by `JwtAdapter.generateToken()`.
</Note>

## How the gateway validates tokens

The gateway uses `JwtValidatorAdapter`, which parses the signed claims using the JJWT library:

```java
// gateway/.../JwtValidatorAdapter.java
@Override
public Map<String, Object> validateToken(String token) {
    try {
        Claims claims = Jwts.parser()
                .verifyWith(key)
                .build()
                .parseSignedClaims(token)
                .getPayload();
        return Map.copyOf(claims);
    } catch (JwtException e) {
        throw new IllegalArgumentException("Invalid or expired token", e);
    }
}
```

The filter extracts `sub`, `role`, and `userId` from the returned claims map and builds the mutated request. Any `JwtException` (tampered signature, expired token) causes the filter to return `401 Unauthorized` immediately.

## Public vs. protected routes

The gateway's `isPublicPath()` logic applies the following rules before attempting JWT validation:

| Condition | Requires auth? |
|---|---|
| `OPTIONS` (CORS preflight) | No |
| `/api/auth/**` (except `/api/auth/motels/**`) | No |
| `/actuator/**` | No |
| `GET /api/motels`, `GET /api/rooms`, `GET /api/services` | No |
| Everything else | Yes |

## Token expiration

The token lifetime is controlled by the `JWT_EXPIRATION` environment variable (milliseconds).

| Variable | Default | Description |
|---|---|---|
| `JWT_EXPIRATION` | `86400000` | Token lifetime in ms — 86 400 000 ms = 24 hours |

To set a shorter lifetime for higher-security environments:

```bash
JWT_EXPIRATION=3600000   # 1 hour
```

## Password reset flow

<Steps>
  <Step title="Request a reset link">
    ```bash
    POST /api/auth/reset-password-request
    Content-Type: application/json

    {"email": "alice@example.com"}
    ```

    The user management service sends a time-limited reset link to the provided email address via the notification service.
  </Step>
  <Step title="Submit the new password">
    ```bash
    POST /api/auth/reset-password
    Content-Type: application/json

    {
      "token": "<reset-token-from-email>",
      "newPassword": "newS3cr3t!"
    }
    ```
  </Step>
</Steps>

<Warning>
  Reset tokens are short-lived and single-use. Submitting an expired or already-used token returns `400 Bad Request`.
</Warning>
